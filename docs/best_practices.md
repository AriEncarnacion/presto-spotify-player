# Best Practices for MicroPython Development on Pimoroni Presto

## Overview

[Pimoroni **Presto** is a 4” touchscreen microcontroller (RP2350-based) with Wi-Fi, Bluetooth, and onboard peripherals (RGB LEDs, piezo speaker, etc.)](https://www.cnx-software.com/2025/04/14/pimoroni-presto-raspberry-pi-rp2350-4-inch-wireless-desktop-touch-display/#:~:text=Pimoroni%20Presto%20,piezo%20speaker%2C%20a%20Qwiic%2F). It runs a custom MicroPython (v1.26) firmware tailored for Presto (firmware v1.0.0 as of 2025). Developing apps for Presto is similar to Pico/Pico W development but with additional **Presto-specific libraries** for the display, touch, Wi-Fi, and more. Below are best practices and common pitfalls in key areas (Wi-Fi, display/graphics, vector drawing, memory management, and general optimization) to help you write efficient and robust Presto MicroPython apps.

## Wi-Fi Connectivity

**Use secrets.py for credentials:** Presto’s Wi-Fi library expects your SSID and password in a file called `secrets.py` on the device. Ensure this file exists on the Presto’s filesystem and contains `WIFI_SSID = "YourNetwork"` and `WIFI_PASSWORD = "YourPassword"` (with quotes). If these variables are missing or empty, presto.connect() will throw an error (e.g. *“ValueError: secrets.py: WIFI_SSID is empty!”* ￼[ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=File%20,08%3B%20Presto) [ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=%E2%80%A6is%20there%20actually%20a%20file,just%20called%20%E2%80%9CSSID%E2%80%9D%20and%20%E2%80%9CPassword%E2%80%9D%E2%80%A6)). A common pitfall is editing secrets.py only on your PC – you must **upload it to the Presto**. Using Thonny’s File menu (or similar IDE), save the edited `secrets.py` directly to the Presto’s flash (shown as “Raspberry Pi Pico” in Thonny) [ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=In%20Thonny%2C%20once%20you%20have,are%20stored%20on%20the%20Pico) 

**One-line connect:** The Presto MicroPython firmware provides an easy connect method. After initializing Presto, simply call **presto.connect()** to join the Wi-Fi network (assuming `secrets.py` is set up). All the network setup is handled under the hood for you ￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=A%20stylish%20way%20of%20displaying,Real%20Time%20Clock) 
For example:

```python
from presto import Presto  
presto = Presto()  
presto.connect()  # Connects to Wi-Fi using credentials in secrets.py
```

On successful connection, you can use MicroPython networking as usual (e.g. send requests). The Presto firmware uses an asynchronous approach internally for Wi-Fi, so this call should not freeze your app unnecessarily ￼[ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=File%20,py%3A%20WIFI_SSID%20is%20empty%21%20%04).

**Manual connection (advanced):** If you need more control or troubleshooting, you can use MicroPython’s network library directly instead of `presto.connect()`. For example, to connect with status feedback and timeout:

```python
import network, time
from secrets import WIFI_SSID, WIFI_PASSWORD

wlan = network.WLAN(network.STA_IF)
wlan.active(True)
wlan.connect(WIFI_SSID, WIFI_PASSWORD)
max_wait = 30  # 30 seconds timeout
while max_wait > 0:
    if wlan.status() < 0 or wlan.status() >= 3:  # <0 = error, >=3 = connected
        break
    max_wait -= 1
    print("Waiting for Wi-Fi connection...")
    time.sleep(1)
if wlan.status() != 3:
    raise RuntimeError("Network connection failed")
print("Connected to Wi-Fi:", wlan.ifconfig())
```

This snippet (adapted from community code) actively waits for connection and prints the IP on success [ref](https://forums.pimoroni.com/t/presto-wifi/26932#:~:text=wlan,Fi%20connection...%22%29%20time.sleep%281)￼. Using the network module directly can help you implement custom retry logic or check `wlan.status()` codes (e.g. to distinguish errors). It’s also good practice to set your Wi-Fi country code for regulatory compliance using `rp2.country('US')` (or your country code) before connecting ￼[ref](https://forums.pimoroni.com/t/presto-wifi/26932#:~:text=import%20time%20import%20network%20import,from%20secrets%20import%20WIFI_SSID%2C%20WIFI_PASSWORD).

**Network error handling:** Always handle potential network errors. For example, fetching internet data (using `urequests/requests` or NTP time sync) may fail due to connectivity or server issues. Wrap such calls in try/except and provide fallback behavior or user messages. The default Word Clock example, for instance, will only print “unable to get time” if NTP fails, so you might improve it by catching exceptions around `ntptime.settime()` to diagnose the issue [ref](https://forums.pimoroni.com/t/presto-upgrade-bricked-my-wi-fi/28151#:~:text=ahnlak%20%20August%2010%2C%202025%2C,9%3A49am%20%206)￼. In general, check that `presto.connect()` actually succeeded (it returns a network interface object on success) and possibly verify with `wlan.isconnected()` or a quick ping (e.g. DNS lookup or HTTP request) before proceeding.

**Keep Wi-Fi usage non-blocking:** Presto’s Wi-Fi is designed to work in the background, but if you perform a long network operation in the main thread, your UI might freeze until it completes. Consider using MicroPython’s **uasyncio** for concurrent tasks (e.g. update display while waiting for network). The Presto library itself leverages asyncio (as seen in the traceback calling `asyncio.run_until_complete` within `presto.connect()`[ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=File%20,py%3A%20WIFI_SSID%20is%20empty%21%20%04)￼), so integrating your code with asyncio (async await calls for network) can yield smoother performance. If not using asyncio, at least give the system time to breathe (e.g., small `time.sleep()` or yielding to scheduler) in long loops waiting for data.

## Display and Graphics

**Drawing with PicoGraphics:** Presto’s display (480×480 LCD) is driven by Pimoroni’s PicoGraphics library, wrapped in the Presto MicroPython module. When you create a `Presto()` object, it sets up a `presto.display` instance for drawing. You can use display methods to draw shapes, text, pixels, etc. (refer to the PicoGraphics docs for full API [ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=You%20can%20find%20a%20full,functions%20in%20the%20PicoGraphics%20documentation)￼). After drawing, **remember to update the screen** by calling `presto.update()`. This flushes the off-screen framebuffer to the physical display [ref](https://forums.pimoroni.com/t/presto-graphics-vs-picographics-oh-dear/27097#:~:text=Well%2C%20it%20is%20display,update%20if%20you%20import%20presto)￼. A common mistake is to draw on `presto.display` and forget to call `presto.update()` – no changes will show on screen. (If you directly use `PicoGraphics` without the Presto wrapper, you’d call `display.update()` instead, but when using `Presto` always call `presto.update()` [ref](https://forums.pimoroni.com/t/presto-graphics-vs-picographics-oh-dear/27097#:~:text=Well%2C%20it%20is%20display,update%20if%20you%20import%20presto)￼.)

**Double-buffering and resolution:** By default, Presto runs in a lower resolution buffered mode for smoother updates. The Presto constructor has a `full_res` option – if `full_res=False` (default), the library uses a lower effective resolution with double buffering (two drawing layers) for the 4” screen; if `full_res=True`, it uses the full 480×480 resolution but only a single buffer by default [ref](https://forums.pimoroni.com/t/presto-graphics-vs-picographics-oh-dear/27097#:~:text=27,if%20layers%20is%20None)￼. The lower-res mode improves performance and reduces flicker by drawing on a back buffer then swapping buffers, at the cost of some sharpness. If your app needs **crisp detail (e.g. small text, fine images)**, enable `full_res=True` for 480×480 rendering. Just be aware that full resolution uses more memory and may refresh slightly slower (due to pushing more pixels). On the other hand, if you only need large text or simple graphics, the default mode can be more efficient. Test both modes to see the performance vs quality trade-off for your app.

**Avoid flicker and tearing:** Even with double buffering, you should clear or redraw the entire area that changes each frame. For example, if you update text or moving graphics, clear the old text or draw a background rectangle before drawing new text, to avoid ghosting. The PicoGraphics API provides convenient methods like `display.clear()` (to clear the whole frame) or drawing filled rectangles to overwrite old content. It’s usually easiest to redraw the full screen each frame (clear, then redraw everything needed, then `presto.update()`), given Presto’s decent speed and memory, unless your scene is very complex.

**Blocking operations and flicker:** Be cautious about doing heavy blocking operations during display updates. The Presto’s display is refreshed by a secondary core/process, but if your main MicroPython code performs certain blocking I/O on the filesystem or similar, it can interfere. A real-world example: a developer noticed severe screen flicker whenever they wrote to a log file while updating the display [ref](https://forums.pimoroni.com/t/presto-display-flickers-on-update/27653#:~:text=when%20updating%20the%20display,video%20to%20show%20the%20problem)￼ ￼[ref](https://forums.pimoroni.com/t/presto-display-flickers-on-update/27653#:~:text=And%20I%E2%80%99ve%20figured%20out%20what,the%20problem%20was). Opening/writing a file (especially on the internal flash) during a screen refresh caused visible tearing. The solution was to avoid file writes in the critical update loop – once they disabled the logging, the display updates became clean ￼[ref](https://forums.pimoroni.com/t/presto-display-flickers-on-update/27653#:~:text=And%20I%E2%80%99ve%20figured%20out%20what,the%20problem%20was). **Best practice:** do not write to flash or perform slow operations in the tight loop of updating the UI. If you need to log data, consider batching it: e.g. collect data in a buffer and write to a file after drawing (or at longer intervals). Similarly, large computations should be offloaded or broken into smaller chunks so the display can refresh smoothly in between.

**Using layers:** Presto’s PicoGraphics supports multiple layers (off-screen drawing surfaces) which can be composited. In fact, as noted, the default mode uses 2 layers internally. You can explicitly request a certain number of layers via `Presto(layers=N)`. This can be useful if you want, say, a static background layer and a dynamic foreground layer (to avoid redrawing the background each time). However, more layers mean more memory usage and complexity. The Presto’s RP2350 has 8MB PSRAM, which can hold multiple 480×480 framebuffers, **but be mindful of memory** (each full-res layer is ~450KB for 16-bit color). Use layers only if needed. Also note that drawing **vector graphics** on layers other than 0 is not directly supported in the current PicoVector API (no `set_layer` function yet) [ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=Is%20there%20a%20way%20to,yet)￼ – a known limitation as of firmware 1.0.0. So, if you plan to use advanced layering with text, you might stick to PicoGraphics text for secondary layers, or manually copy images between layers as a workaround ￼[ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=1,af%20fonts).

**Touch UI considerations:** Presto’s touch screen (capacitive) can be read via presto.touch. Pimoroni’s library provides a simple `Button(x, y, w, h)` class (in the `touch module`) to define touch regions [ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=Using%20touch) [ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=touch%20%3D%20presto)￼ ￼. Poll the touch input (`touch.poll()`) each loop to update touch state, and then check `button.is_pressed()` for each onscreen button [ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=if%20button_1,update)￼. When designing touch UIs, draw visual feedback for touches (e.g. highlight buttons on press as in the example) and call `presto.update()` after updating the UI. Common pitfalls: forgetting to call `touch.poll()` (then `.is_pressed()` won’t update), or not redrawing the button state (so the screen doesn’t show the press). Also, if touches seem misaligned, remember the coordinates you use for buttons must match the coordinates of your drawings. The touch coordinates map 1:1 to the display pixels (in full_res mode; in lower res mode, they should still correspond to the scaled coordinates, since `FT6236(full_res=...)` is handled by Presto) [ref](https://forums.pimoroni.com/t/presto-graphics-vs-picographics-oh-dear/27097#:~:text=23,Touch%20Input)￼. In short, treat touch like another input (like a button) and update your display accordingly.

## Vector Graphics and Text

**When to use PicoVector:** PicoVector is Pimoroni’s vector graphics module, enabling scalable text and shapes using outline fonts (`.af` files, “Alright Fonts”). Use it for large or stylistic text, smooth curves, and arbitrary shapes. Vector graphics are rendered in software and are slower than the built-in bitmap drawing functions ￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=Vector%20fonts%20are%20slower%20than,great%20for%20large%2C%20beautiful%20text). The benefit is you can choose from many fonts and scale/rotate text without jagged edges or pixelation. If your app requires big, beautiful headings or custom fonts, PicoVector is ideal; if you just need small labels or numbers, the default bitmap font (via `display.text()`) is much faster and uses less memory ￼[ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=1,af%20fonts).

**Using vector fonts:** Presto’s firmware includes a few default vector fonts (e.g. `Roboto-Medium.af`). To use one, load it into a PicoVector instance and set size. For example:

```python
from picovector import PicoVector, ANTIALIAS_BEST
vector = PicoVector(presto.display)
vector.set_antialiasing(ANTIALIAS_BEST)  # or ANTIALIAS_FAST for speed
vector.set_font("Roboto-Medium.af", 90)  # Font file must be on device
vector.text("Hello!", 10, 150, max_width=300)
presto.update()
```
This will draw “Hello!” at roughly 90px font size, anti-aliased for quality [ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=MAGENTA%20%3D%20display,create_pen%280%2C%200%2C%200)￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=display.set_pen%28BLACK%29%20vector.text%28,update) ￼. Ensure the font file is present on the device (the Presto “launcher” firmware usually installs Roboto by default ￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=Assuming%20you%20have%20the%20launcher,be%20present%20on%20your%20device), but if you want others, you can add converted `.af` files). Keep in mind vector text rendering can be intensive – use simpler antialiasing (`ANTIALIAS_FAST`) or smaller sizes if you need to speed it up, and avoid redrawing large vectors every frame if not necessary.

**Vector shapes:** PicoVector isn’t just for text – you can draw shapes like polygons, arcs, circles, lines, etc., by constructing Polygon objects and using methods like `polygon.arc(x, y, radius, start_angle, end_angle, thickness)` or `polygon.circle(x, y, radius, thickness)` ￼ ￼. After defining the shape, use `vector.draw(polygon)` [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=astart%20%3D%2090%20aend%20%3D,circle%28xpos%2Cypos%2Cr%2C3) [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=arc_a%20%3D%20Polygon%28%29%20arc_a,circle%28xpos%2Cypos%2Cr%2C3) to render it to the screen buffer [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=)￼. A gotcha here is that complex shape drawing can be slow if done repeatedly in a loop. If you need animation, consider drawing simpler shapes or only updating small parts of the shape. In one case, drawing multiple arcs in a loop caused the program to lock up when run repeatedly in Thonny (likely due to a bug or memory issue) [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=I%20was%20playing%20with%20PicoVector,flashed) [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=for%20i%20in%20range,arc%202)￼ ￼. This was alleviated by resetting the device between runs and was potentially related to resource cleanup. **Best practice:** If you draw vector shapes in a loop, try to **reuse objects** instead of creating new `Polygon()` each time. For example, create a Polygon once, then modify it or redraw it as needed. This avoids constant allocation and garbage collection mid-loop. Also call `gc.collect()` after heavy drawing loops (see Memory section) to free any leftover allocations. In the arc-drawing example, moving the `arc_a = Polygon()` outside the loop improved stability (fewer iterations before slowdown) [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=Resetting%20the%20Presto%20before%20doing,which%20it%20effectively%20is)￼, suggesting that frequent allocation was a factor.

**No layer support (yet):** As mentioned, PicoVector currently draws to the default layer only – you cannot directly target a secondary layer for vector output [ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=Is%20there%20a%20way%20to,yet)￼. If you want to overlay vector text on an animated background without redrawing it every frame, one workaround is to render the text once and save it. For instance, draw the vector text to the screen, then copy that portion of the framebuffer to an in-memory buffer (or use it as a static image). Another simpler approach: use vector for a one-time render of a fancy title or icon at startup, then switch to bitmap graphics for dynamic content ￼[ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=1,af%20fonts) [ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=Thanks%2C%20I%20tried%200,anything%20new%20on%20this%20front) ￼. A community member did this to achieve 64 FPS animations with static vector text: draw the text once, then handle moving graphics with faster routines, essentially implementing their own layering in code ￼[ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=I%20ended%20up%20going%20with,Pretty%20happy%20with%20the%20results). Until official layer support is added to PicoVector, these strategies can help balance performance and aesthetics.

**Fonts and assets:** If you need additional vector fonts, Pimoroni provides a pack of converted fonts and a Python script to convert `TTF/OTF` to `.af` format ￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=More%20fonts%3F). Keep the font files as small as possible (they’re outlines, so typically a few tens of KB each) and only include those you use to save memory. Bitmap sprites or icons can be drawn using `display.image()` or by converting images to Python byte arrays – but note that MicroPython can struggle with very large images. Prefer JPEG decoding for large images (see Memory section), or design your UI with simple shapes and text rather than large bitmaps to stay within memory and speed limits.

## Memory Management

**Leverage Presto’s RAM wisely:** Presto has **520KB SRAM + 8MB PSRAM**, giving MicroPython a much larger heap than a standard Pico ￼[ref](https://thepihut.com/products/pimoroni-presto?srsltid=AfmBOooWAXtsCjJ6jHl4YUM5UW_R82TmZfs72o5SwYIyocf3-5-bZyhk#:~:text=Pimoroni%20Presto%20,). This is great for holding images, buffers, and data, but it can lull you into using memory inefficiently. MicroPython’s garbage collector (GC) runs on-demand (typically when an allocation fails or on explicit call) and uses a **mark-and-sweep** algorithm [ref](https://docs.micropython.org/en/latest/develop/memorymgt.html#:~:text=Memory%20Management%20%E2%80%94%20MicroPython%20latest,traverses%20the%20heap%20marking)￼. Large heaps can become fragmented or slower to garbage-collect if you create and destroy many objects. Best practices for memory:
* **Avoid excessive allocation in loops:** Try not to create lots of new objects (strings, lists, polygons, etc.) every iteration of a tight loop. This can fragment the heap and eventually lead to `MemoryError` even if total free memory is sufficient (due to fragmentation). Instead, allocate needed buffers or objects once and reuse them. For example, if you need a buffer for incoming data, allocate a bytearray up front. If you are parsing sensor or API data repeatedly, reuse the same list or clear it instead of creating a new list each time.
* **Manual garbage collection:** It can be helpful to manually trigger `gc.collect()` at strategic points, such as after loading a large image or after a loop that created a lot of temporary objects. This forces a cleanup of unused memory. One community member recommended implementing “memory clean-up strategies” to address suspected memory leaks or buffer overruns when drawing complex graphics [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=Another%20root%20cause%20could%20be,consumption%20of%20the%20polygon%20function)￼. In practice, calling `gc.collect()` after a big task (e.g., after decoding a large JSON or after drawing a complex frame) can proactively free memory for the next step. This can prevent cumulative fragmentation from crashing your long-running app.
* **Use streaming for large data:** If you need to handle large files or network responses, process them in streaming fashion rather than reading everything into memory. For example, to display a large JPEG image on the screen, use the jpegdec module to decode it directly to the framebuffer in chunks, rather than loading the whole image into RAM. The Presto MicroPython firmware includes an optimized JPEG decoder (`jpegdec.JPEG`) for this purpose [ref](https://forums.pimoroni.com/t/presto-and-large-jpg-photos/27182#:~:text=from%20presto%20import%20Presto%20import,display) [ref](https://forums.pimoroni.com/t/presto-and-large-jpg-photos/27182#:~:text=presto.update%28%29%20,JPEG%28display)￼ ￼. As shown in Pimoroni’s examples, you can do:

```python
import jpegdec
j = jpegdec.JPEG(presto.display)
j.open_file("myphoto.jpg")
j.decode(0, 0, jpegdec.JPEG_SCALE_FULL)  # draw at (0,0) full size
presto.update()
```

This decodes the image in situ. Loading a full 480×480 image uncompressed could use hundreds of KB, so decoding on the fly is both memory- and speed-friendly. Likewise for network data: if downloading text (e.g. JSON or CSV), read it in line-by-line or in small chunks and handle each chunk, rather than doing one giant response.content for a huge payload. MicroPython can handle surprisingly large data given Presto’s RAM, but you must still code carefully to avoid big spikes in memory usage.

* **Freeing unused resources:** If your app dynamically imports modules or loads assets that aren’t needed later, consider unloading them to free memory. You can del large lists or objects when done and call `gc.collect()`. MicroPython doesn’t automatically unload modules, so if you import a large module (or one of the example files) and no longer need it, you can remove references to it. However, be careful with this, as you cannot unload a module completely without resetting the device. Often it’s better to structure the program so everything needed stays loaded.
* **PSRAM performance note:** The external 8MB PSRAM is a bit slower than internal RAM ￼[ref](https://blog.adafruit.com/2025/04/28/improving-garbage-collection-time-in-circuitpython/#:~:text=Improving%20garbage%20collection%20time%20in,memory%20related%20problems%20more%20obvious). You likely won’t notice this in Python-level code except in very memory-intensive operations, but extremely large data structures might have slightly slower access. The trade-off for having the memory is worth it; just avoid doing something like iterating byte-by-byte over a 5MB bytes object in Python – that would be slow regardless of PSRAM. If you must manipulate large buffers, consider using MicroPython’s `array` or `memoryview` to do it in C when possible.

## Performance and Optimization

**Use hardware features and C-optimized routines:** Wherever possible, use the provided libraries (written in C under the hood) instead of “pure Python” implementations. For example, use `presto.display.rectangle(x,y,w,h)` to draw a filled rectangle instead of setting individual pixels in Python – the former is optimized and much faster. The same goes for drawing text, shapes, etc. The Presto firmware’s built-ins (PicoGraphics, PicoVector) are your friends for performance. If you need to do heavy computations (like complex math or data parsing), try to use Python’s built-ins or vectorized approaches. The RP2350 has a dual-core Cortex-M33 at up to 150 MHz [ref](https://thepihut.com/products/pimoroni-presto?srsltid=AfmBOooWAXtsCjJ6jHl4YUM5UW_R82TmZfs72o5SwYIyocf3-5-bZyhk#:~:text=Pimoroni%20Presto%20,)￼, but Python can only push it so far. In extreme cases, MicroPython does support native code emitters (via `@micropython.native` or `@micropython.viper` decorators) – these can speed up math-heavy functions significantly, though you must carefully follow their constraints. This is an advanced optimization and usually not needed unless you profile your code and find Python is too slow in a specific routine.

**Concurrency:** Presto’s display and Wi-Fi operations involve concurrency. The display driver runs on core1 (a separate core) to refresh the screen without bogging down your code [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=m_new_class%28ST7701,reset%20your%20MicroPython%20device)￼, and Wi-Fi can operate asynchronously. To take advantage of this, structure your program to allow parallelism: e.g., do not put the device in `time.sleep(10)` if you can help it – that would stall everything. Instead, if waiting for something, consider shorter sleeps in a loop, or better, an `asyncio` sleep so other tasks (like Wi-Fi or screen refresh) can run. MicroPython’s **`_thread`** module can also spawn a second thread on RP2040’s second core, **but be cautious:** Presto’s internal firmware is already using the second core for the screen, so attempting to use `_thread` may conflict or not be supported. The safer route is to use asyncio cooperative multitasking. For instance, you could have one coroutine fetch data from an API periodically, and another coroutine update the display clock every second, running concurrently. This way the Wi-Fi latency doesn’t freeze the UI completely.

**Power supply and stability:** It’s worth noting that running the Presto at full tilt (Wi-Fi active, screen updating, RGB LEDs on) draws a decent amount of current (often ~200mA or more) ￼[ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=It%E2%80%99s%20on%20a%20USB,2A). If the USB port or hub you use cannot supply sufficient stable current, you might encounter weird behavior (freezes, spontaneous resets). If you see the Presto locking up especially during high activity, ensure it’s on a reliable USB power source. One user found that connecting Presto directly to a powered USB port (instead of a bus-powered hub) resolved mysterious lock-ups during graphics demos [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=m1cr0py7h0n%20%20January%207%2C%202025%2C,10%3A49pm%20%204) [ref](https://forums.pimoroni.com/t/presto-picovector-issue/27032#:~:text=If%20yes%2C%20I%20still%20doubt,the%20port%20was%20the%20solution%E2%80%A6)￼ ￼. So, if you run into unexplained crashes, consider power as a potential culprit – use a good quality USB cable and a port that can supply 5V/1A (though Presto shouldn’t need nearly that much, overhead is good).

**Firmware updates:** Pimoroni continues to improve the Presto firmware. Upgrading to the latest firmware can bring performance enhancements and bug fixes (for example, a release in early 2025 improved PicoVector speed and fixed wireless code by rebasing on upstream MicroPython [ref](https://forums.pimoroni.com/t/picovector-layer-support/27498#:~:text=Release%20Version%200)￼). Keep an eye on the [Presto releases page](https://github.com/pimoroni/presto/releases) and update if you’re running an older version. Just remember to back up your files before flashing a new UF2. If you use the “-with-examples” firmware UF2, it will **erase the filesystem** and replace it with example files [ref](https://github.com/pimoroni/presto#:~:text=There%20are%20two%20choices%2C%20a,which%20includes%20everything%20in%20examples)￼. To avoid losing your work, save your Python scripts to your computer or use the regular firmware (which preserves the existing filesystem). After any firmware update, double-check that `secrets.py` still exists and has your credentials (re-add it if needed, since a with-examples flash might reset it) ￼[ref](https://forums.pimoroni.com/t/presto-upgrade-bricked-my-wi-fi/28151#:~:text=%E2%80%9Cunable%20to%20get%20time) [ref](https://forums.pimoroni.com/t/presto-upgrade-bricked-my-wi-fi/28151#:~:text=,whichever%20NTP%20server%20it%20picks)￼.

**Integrating with the launcher:** Presto comes with a built-in app launcher (main menu) that runs on boot. If you develop your own app and want it to appear in the launcher, simply save your script as a .py file in the root of the Presto’s drive. The launcher will list it automatically ￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=If%20you%20save%20a%20,like%20for%20the%20Cheerlights%20example). You can customize how it appears by adding special comments at the top of your file for the app’s **name, description, and icon**. For example, in Pimoroni’s Cheerlights demo, the top of the file contains an `# ICON` comment with a list of coordinate pairs defining a vector icon shape ￼[ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=top%20of%20your%20example,like%20for%20the%20Cheerlights%20example). You can do the same:

```python
# NAME: My Cool App
# DESCRIPTION: Does something neat with Presto
# ICON: [[(path coordinate pairs...)]]
```

The `ICON` is defined as a list of polygon paths (in normalized coordinates) – you can borrow the format from Pimoroni’s examples and adjust, or stick to `NAME` and `DESCRIPTION` if you don’t want a custom icon. Including these comments will make your app show up in the menu with a nice title and icon. When testing, you might temporarily disable the launcher (e.g., by renaming `main.py`) to have your code run on boot, but for normal use it’s convenient to launch from the menu.

**Development tools:** For the best experience, use an IDE that supports MicroPython file operations. **Thonny** is highly recommended by the community [ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=newtonheath%20%20January%2015%2C%202025%2C,11%3A58am%20%204)￼ [ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=I%20use%20Thonny%20for%20ages,swift%20and%20lightweight%E2%80%A6all%20you%20need)￼. In Thonny, select the MicroPython (Raspberry Pi Pico) interpreter, connect to Presto, and use **View → Files** to see the device’s file system ￼[ref](https://forums.pimoroni.com/t/help-with-wifi-presto-connect-on-presto/27088#:~:text=In%20Thonny%2C%20once%20you%20have,are%20stored%20on%20the%20Pico). From there you can easily transfer your `.py` files and assets to the device. Thonny also lets you interrupt a running program (which is useful because Presto auto-runs its launcher on boot – Thonny may need to stop it to get a REPL) [ref](https://learn.pimoroni.com/article/getting-started-with-presto#:~:text=,you%20that%20Presto%20is%20talking)￼. If you use other editors (Mu, VS Code with mpremote, etc.), ensure you know how to upload files and access the REPL. Always **stop the currently running program** on Presto before trying to run another – for instance, press the Stop button in Thonny to interrupt the launcher. This prevents weird conflicts like multiple scripts running or the device being busy (which can manifest as a *“device is busy or not responding”* error).
